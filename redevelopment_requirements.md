# 단독 네이버 카페 키워드 노출 모니터링 서비스 (N-Monitor) 통합 요구사항 정의서

본 문서는 네이버 카페에 특정 브랜드가 포함된 게시물이 정상적으로 노출되고 있는지 확인하고 지속 모니터링할 관리용 대시보드를 구축하기 위한 요구사항을 정의합니다. 새로 합류할 개발자에게 전체 시스템 스펙 및 UI 방향성을 명확히 전달하기 위해 작성되었습니다.

---

## 1. 프로젝트 개요

- **목적**: 등록된 네이버 검색 키워드에 대해 특정 브랜드가 포함된 카페 게시물이 노출되고 있는지 자동으로(또는 수동으로) 확인하고, 상태(진행 현황) 및 순위를 관리하는 시스템 구축.
- **아키텍처**: 프론트엔드(Client)와 백엔드(API & Crawler)의 완전한 분리로 구성.

---

## 2. 주요 기능 및 백엔드 (Backend) 요구사항

### 2.1. 데이터 관리 (CRUD)
- **키워드 등록**: 키워드(다중 입력 가능), 브랜드명, 담당자, 업체명 정보를 입력하여 모니터링 대상을 등록.
- **조회/필터링**: 전체 등록 데이터를 표(Table) 형태로 조회하며 특정 상태, 담당자, 키워드, 혹은 상태 탭 필터 등으로 리스트 필터링.
- **수정/삭제**: 등록된 항목의 상태값 등 세부 정보를 수정하거나 항목을 삭제 (수정/삭제 시 로깅).

### 2.2. 상태 워크플로우 관리
생성된 각 모니터링 작업은 다음의 상태값을 가지며, 수동 또는 크롤링 결과에 따라 변경됨:
1. `대기 (WAITING)`
2. `최초 노출 (FIRST_EXPOSED)`
3. `미노출 (NOT_EXPOSED)`
4. `건바이 성공 (SUCCESS_24H)`
5. `미노출 AS (AS_NEEDED)`
6. `판단불가 (UNCLEAR)`
7. `정산 대기 (SETTLEMENT_WAIT)`
8. `입금 완료 (PAID)`

### 2.3. 크롤링 엔진 (동기/비동기 혼합)
- **검색 결과 긁어오기**: 네이버 통합 검색결과(검색조회후 첫번째 화면)에서 특정 키워드를 검색하여 상위 노출되는 URL(cafe.naver.com 패턴) 파싱. 절대 다른 탭(블로그, 카페)의 결과는 가져오지 않는다.
- **게시글 본문/댓글 확인**: 파싱된 개별 게시글에 접근(iframe 처리 등 포함)하여 페이지 댓글 내(text_content)에 **'브랜드명'**이 존재하는지 텍스트 검색. 네이버 카페 글 읽기 권한이 필요한 글은 접근하지 않는다.
- **결과 산출**: 브랜드명이 포함된 댓글을 발견하면 해당 글의 '현재 순위(예: 3/12)'와 URL을 데이터베이스에 저장. 처음 발견된 URL은 `initial_url`로 저장되고, 추후 검사 시 URL이 바뀌었는지 여부도 함께 추적(`2/9(유지중)` 또는 `2/11(URL바뀜)`).

### 2.4. 노출 확인 트리거 및 스케줄러
- **수동 확인(Manual Check)**: 사용자가 버튼(순위조회 등)을 눌렀을 때 특정 키워드(또는 전체 키워드)의 노출 여부를 실시간으로 검사.
  - 이 때 진행률 및 결과를 단일 페이지 내에서 실시간으로 프론트엔드로 전달해야 함 (예: SSE(Server-Sent Events) 또는 WebSocket 활용).
- **자동 스케줄링**: 매일 정해진 시간변수(예: 오전 8시 50분)에 자동으로 전체 키워드의 현재 노출 상태 및 순위를 검사하여 DB 업데이트 (24H 자동 모니터링 ON/OFF 기능 제공).

### 2.5. 이력(Log) 관리
- 각각의 키워드에 대해 상태(Status)가 변경되거나 주요 액션이 일어날 때마다 `이전 상태 -> 변경된 상태`, `변경자`, `메시지`, `변경일시`를 로그로 기록하고 UI에서 조회 또는 백엔드에서 관리가 가능해야 함.

### 2.6. 백엔드(Backend) 세부 설정 스펙
- **기술 스택 (권장)**: Python (FastAPI) 또는 Node.js (NestJS/Express). (크롤링 등 고려시 Python/FastAPI나 Node.js 환경이 유리)
- **Database**: 범용 가능한 SQLite 또는 MySQL/PostgreSQL 등 관계형 DB 적용.
- **API 설계**: RESTful API 방식으로 프론트엔드와 완벽하게 분리.
  - `GET /api/tasks`, `POST /api/tasks`, `PUT /api/tasks/:id`, `DELETE /api/tasks/:id`
  - `GET /api/logs?taskId=X` (상태 변경 로그)
  - `POST /api/check-exposure` (비동기 크롤링 작업 큐잉)
  - `GET /api/events/:job_id` (SSE 스트리밍 - 크롤링 진행 상태 전달)
- **크롤러 시스템 구축 주의사항**:
  - 네이버 검색 시 IP 차단, 캡차 등 어뷰징 방지 로직 우회 프로세스 필요 (Playwright 스텔스 모드, 랜덤 딜레이, User-Agent 변경 등).
  - 브라우저 바이너리(Chromium) 관리 및 동시성(Concurrency) 관리. 불필요한 메모리 누수가 발생하지 않도록 ThreadPool 또는 비동기 큐잉 시스템 구축.
- **동시성 처리 및 성능 보장**: 백엔드는 여러 개의 키워드 조회가 동시에 들어와도 데드락이나 병목이 발생하지 않도록 설계.

## 3. 프론트엔드 (Frontend) 요구사항

해당 서비스는 제공된 디자인(다크 테마 기반, 퍼플/블루 포인트 컬러 적용)에 부합하는 UI/UX로 구현되어야 합니다. 사용자는 모두 단일 페이지에서 진행 상황을 파악하고 결과를 도출할 수 있어야 합니다.

- **기술 스택 (권장)**: React, Next.js (App Router), TypeScript, Tailwind CSS (또는 Styled-components 등)
- **전체 레이아웃 및 헤더 구조**:
  - GNB(글로벌 네비게이션 바): 서비스 로고(N-Monitor), 메인 메뉴 탭('메인 대시보드', '정산 아카이브'), 주기적 자동 스케줄러 활성화/비활성화 상태 뱃지 및 토글 기능('24H 자동 모니터링 ON').
- **상태 요약 카드 (Summary Cards) 영역**:
  - 상단에 중요 상태별 건수를 한눈에 파악할 수 있는 카드 섹션 배치.
  - 제공 카드 지표 수치 항목: `전체 활성`, `대기`, `최초 노출`, `건바이 성공`, `미노출 AS`, `판단불가`
- **입력 폼 영역 (키워드 등록)**:
  - 영역 상단에 [접기 ▲] / [펴기 ▼] 조작 가능한 토글 버튼 제공.
  - 입력 필드 구성:
    - `키워드`: 줄바꿈으로 다중 키워드 입력 가능한 textarea 형태.
    - `브랜드명`: 텍스트 인풋 박스.
    - `담당자`: 옵션을 선택할 수 있는 Select Box 드롭다운.
    - `업체`: 옵션을 선택할 수 있는 Select Box 드롭다운.
    - 하단 제출 버튼: [등록] 버튼.
  - 입력 폼 영역에는 `placeholder` 등 예시 텍스트 없이 모두 빈 칸으로 구성되어야 함.
- **데이터 리스트 영역 (모니터링 태스크)**:
  - 우측 혹은 상단에 총 태스크 개수 뱃지 표시(예: 모니터링 태스크 4)와 상태별 필터 기능(버튼 탭 형식) 제공:
    - 탭 구성: `전체`, `대기`, `노출확인`, `성공`, `미노출AS`, `미노출(검색결과없음)`
  - **테이블 컬럼 구조 표기 사항**: 
    1. `등록일` (형식: YYYY.MM.DD HH:mm)
    2. `키워드`
    3. `브랜드명` (가독성을 위한 민트 계열 하이라이트/포인트 색상 텍스트 적용)
    4. `담당자`
    5. `업체`
    6. `상태` (상태값에 따른 고유 색상 뱃지 적용. 예: '최초 노출'은 퍼플 뱃지/흰색 텍스트)
    7. `최초/현재순위` (표기 예시: `1/5 / 1/5(유지중)` 또는 `미노출 / 미노출`)
    8. `24H 예정` (자동 검사 타이머 카운트다운/수치 표시)
    9. `링크` (카페 원문 URL - 클릭 시 이동)
    10. `액션` (개별 조작 버튼: `[순위조회]`, `[재작업]`, `[X 삭제]`)
- **실시간 피드백 및 체감 성능**:
  - 테이블 내 Action(`[순위조회]` 등) 기능 버튼 클릭 시 자동 새로고침(Auto-refresh) 없이 웹소켓 또는 SSE를 통해 "pending -> running -> done/error" 등으로 진행률 및 결과를 실시간 업데이트하여 즉시 표기.

## 5. 품질 및 기타 요구사항
- **의존성 파편화 제거**: Docker 컨테이너화를 고려하거나 명확한 패키지 매니징 환경(requirements.txt / package.json)을 구성하여 신규 개발자가 세팅하기 용이하도록 지원.
- **오류 처리 및 로깅**: 크롤러 등에서 발생하는 예외 현상(네이버 DOM 구조 변경 등)을 빠르게 파악할 수 있도록 체계적인 로깅 구현(Loguru, Winston 등 적용 고려).
- **보안/환경 변수**: DB 정보, API URL, 스케줄러 설정값 등은 모두 환경변수(`.env`)로 분리.

## 6. 상태 전환 흐름 (State Transition Rules)

각 상태는 아래의 조건과 트리거에 의해서만 전환된다. 명시되지 않은 전환은 허용하지 않는다 (단, 수동 재작업 시 `대기`로 리셋은 언제든 허용).

```
[대기 (WAITING)]
  ├─ 크롤링 결과 → 브랜드 발견 → [최초 노출 (FIRST_EXPOSED)]
  ├─ 크롤링 결과 → 브랜드 미발견 → [미노출 (NOT_EXPOSED)]
  └─ 크롤링 중 오류 / 결과 불명확 → [판단불가 (UNCLEAR)]

[최초 노출 (FIRST_EXPOSED)]
  ├─ 최초 노출 확인 시각으로부터 24시간 후 자동/수동 재검사
  │   ├─ 동일 게시글 번호 유지 + 브랜드 발견 → [건바이 성공 (SUCCESS_24H)]
  │   ├─ 게시글 번호 변경됨 → 새로운 [최초 노출 (FIRST_EXPOSED)] 로 리셋 (타이머 재시작)
  │   ├─ 브랜드 미발견 → [미노출 AS (AS_NEEDED)]
  │   └─ 크롤링 오류 → [판단불가 (UNCLEAR)]
  └─ 수동 재작업 → [대기 (WAITING)] (최초순위 확인부터 다시 시작)

[미노출 (NOT_EXPOSED)]
  └─ 수동 재작업 → [대기 (WAITING)] (최초순위 확인부터 다시 시작)

[건바이 성공 (SUCCESS_24H)]
  └─ 사용자 수동 전환 → [정산 대기 (SETTLEMENT_WAIT)]

[미노출 AS (AS_NEEDED)]
  └─ 수동 재작업 → [대기 (WAITING)]

[판단불가 (UNCLEAR)]
  └─ 수동 재작업 → [대기 (WAITING)]

[정산 대기 (SETTLEMENT_WAIT)]
  └─ 사용자 수동 전환 → [입금 완료 (PAID)]

[입금 완료 (PAID)]
  └─ (최종 상태, 이후 전환 없음)
```

**24H 카운트다운 기준**: `최초 노출(FIRST_EXPOSED)` 상태로 변경된 시각을 기준으로 +24시간 후를 자동 재검사 예정 시각(`verified_at`)으로 저장한다. 이 시각이 도래하면 스케줄러 또는 수동 트리거에 의해 재검사가 실행된다.

**건바이 성공 조건 및 기준**:
- `건바이 성공(SUCCESS_24H)`의 기준은 **동일한 게시글 번호**의 게시글이 네이버 검색 결과 1페이지 통합검색 화면에 24시간 유지되며 노출되는 것이다 (댓글에 브랜드명 노출 기준).
- 만약 최초 노출 후 24시간이 되기 전(중간에) 노출되는 게시글 번호가 변경되었다면, 이는 건바이 실패이며 새로운 `최초 노출`로 판단하고 24시간 카운트를 처음부터 다시 시작한다.
- `최초 노출`이나 `미노출` 등에서 `재작업` 버튼을 눌러 `대기` 상태로 리셋한 경우, 기존 이력과 무관하게 처음부터 다시 최초순위 확인 과정을 거쳐야 한다.

**수동 상태 전환 허용 범위**:
- 사용자는 `건바이 성공 → 정산 대기`, `정산 대기 → 입금 완료` 전환을 UI에서 수동으로 수행할 수 있다.
- 그 외 상태에서 `대기`로 되돌리는 재작업은 언제든 허용한다.
- 직접적인 임의 상태 점프(예: `대기 → 건바이 성공`)는 허용하지 않는다.

| 필드명 | 타입 | 설명 |
|---|---|---|
| initial_url | String | 최초 발견 시 저장된 카페 게시글 URL. 이후 URL 변경 추적의 기준점이 된다. |
| verified_at | DateTime | 24H 자동 재검사 예정 시각. `최초 노출` 상태 진입 시각 + 24시간으로 계산하여 저장. |

> **참고**: `url` 필드는 현재 확인된 최신 URL을 저장하고, `initial_url`은 최초 발견 URL을 불변값으로 유지한다. `current_rank` 표기 시 두 URL의 게시글 번호를 비교하여 `(유지중)` 또는 `(URL바뀜)` 접미사를 붙인다.

## 8. 크롤링 예외 처리 명세

크롤러는 다음 상황별로 정의된 동작을 수행해야 한다. 예외를 무시하거나 임의로 처리하지 않는다.

| 상황 | 처리 방법 |
|---|---|
| 검색 결과에 카페 게시글 URL이 하나도 없는 경우 | `미노출(NOT_EXPOSED)` 처리. 메시지: "검색 결과에 카페 게시글 없음" |
| 카페 게시글이 읽기 권한(로그인/가입 필요)으로 접근 불가한 경우 | 해당 게시글은 건너뛰고(skip) 다음 URL 검사. 결과에 미포함. |
| 게시글 접근은 됐으나 브랜드명이 없는 경우 | 해당 게시글을 검사 완료로 처리하되 발견 없음으로 기록. |
| 페이지 로딩 타임아웃 발생 시 | 해당 게시글 skip 후 다음 URL 진행. |
| 전체 크롤링 과정 중 예상치 못한 오류 발생 시 | `판단불가(UNCLEAR)` 처리. 오류 메시지를 로그에 기록. |
| 네이버에 의해 검색이 차단된 것으로 의심되는 경우 (예: 캡차 페이지 감지) | 즉시 크롤링 중단 후 `판단불가(UNCLEAR)` 처리. 관리자가 확인할 수 있도록 로그에 경고 기록. |

## 9. 정산 아카이브 뷰 명세

GNB의 `정산 아카이브` 탭은 `정산 대기(SETTLEMENT_WAIT)` 및 `입금 완료(PAID)` 상태인 태스크만을 별도로 모아서 보여주는 전용 뷰이다. 메인 대시보드 테이블과는 컬럼 구성이 다르다.

- **표시 대상**: `status`가 `SETTLEMENT_WAIT` 또는 `PAID`인 태스크 전체.
- **테이블 컬럼 구성**:
  1. `정산이관일` — `SETTLEMENT_WAIT` 상태로 변경된 일시 (로그에서 추출)
  2. `키워드`
  3. `브랜드명`
  4. `담당자`
  5. `업체`
  7. `상태` — `정산 대기` 또는 `입금 완료` 뱃지
  8. `액션` — `[입금 완료]` (정산 대기 상태일 때만 활성), `[삭제]`
- **필터**: 월별 필터 기능 제공 (예: 2026년 3월 건만 표시).
- **건수 집계**: 해당 월의 정산 대기 건수 / 입금 완료 건수를 헤더에 표시.
- **표시 정보**: 업체별 최종 **입금 완료(PAID) 건수** 집계하여 보여준다.
- **날짜 필터**: 날짜 필터(드롭다운 메뉴)를 제공하여, 특정 기간 내에 업체별로 각각 몇 건이 입금되었는지 한눈에 파악할 수 있도록 한다.
- **구글 시트 연동**: 실제 `입금 완료` 처리된 개별 건의 리스트 목록은 외부 **구글 시트(Google Sheets)에 자동으로 리스팅**되도록 연동한다.

## 10. 담당자 및 업체 옵션 관리

입력 폼의 `담당자`와 `업체` Select Box에 표시할 옵션 목록은 다음과 같이 처리한다.

- **초기 구현**: 프론트엔드 코드에 하드코딩된 배열로 관리하는 것을 허용한다. 별도 관리 페이지 구현은 이번 범위에 포함하지 않는다.
- **옵션 추가/변경**: 프론트엔드 코드를 직접 수정하여 배열에 항목을 추가/제거하는 방식으로 유지보수한다.
- **옵션 구성 예시** (실제 운영 옵션으로 교체 필요):
  - 담당자: 담당자A, 담당자B, 담당자C, 기타
  - 업체: 업체A, 업체B, 업체C, 기타

## 11. 배치(전체 일괄) 조회 UX 명세

단일 태스크 순위조회 외에, 전체 또는 특정 상태의 태스크를 일괄로 조회하는 기능이 필요하다.

- **트리거**: 테이블 상단 또는 헤더 영역에 `[전체 노출 확인]` 버튼 제공.
- **실행 대상**: `대기(WAITING)` 상태인 태스크 전체를 대상으로 한다.
- **진행 방식**:
  - 버튼 클릭 시 전체 대기 태스크 수와 함께 확인 다이얼로그를 표시한다.
  - 확인 후, 각 태스크별 크롤링 작업을 순차 또는 병렬로 실행한다.
  - 전체 진행률(예: `3 / 7 완료`)을 버튼 혹은 별도 영역에 실시간으로 표시한다.
  - 모든 작업 완료 후 테이블을 자동 갱신한다.
- **오류 처리**: 일부 태스크가 실패해도 나머지 태스크 처리는 계속 진행한다. 실패한 태스크는 `판단불가` 처리 후 전체 완료 메시지에 실패 건수를 포함한다.
